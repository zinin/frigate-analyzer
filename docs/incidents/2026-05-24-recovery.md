# Recovery report — 2026-05-24

Продолжение [incident 2026-05-23](2026-05-23-sata-cable-corruption.md). День физической починки железа + чистки btrfs + drop/recreate БД + успешного перезапуска pipeline.

## TL;DR

SATA-кабель на /dev/sda физически заменён. `CRC_Error_Count` зафиксировался на старом значении `1,525,536` и больше не растёт. `ata1 SError` в логе ядра — 0 событий с момента нового boot. `btrfs scrub /` за 5:07 нашёл 75,033 latent corruption на sda3 и **все исправил из mirror** (`Uncorrectable: 0`). `pg_amcheck` показал, что все heap-таблицы целы, но **все индексы recordings + 3 индекса detections + PK object_tracks битые**, плюс **duplicates в `recordings.id`** (corrupted unique constraint не отрабатывал во время инцидента) — обычный REINDEX упал. Принято решение: **drop+recreate frigate_analyzer**, поскольку исторические recordings не критичны (исходные `.mp4` хранятся отдельно на /mnt/data/sdb1, исторический бекfill отключён `DISABLE_FIRST_SCAN=true`). После рестарта контейнера WatchRecordsTask жив, новые `.mp4` индексируются. Команды `/timezone` и `/notifications` после очистки БД требуют, чтобы owner отправил боту `/start` (он не auto-bootstrap-ится из env var — новая открытая issue).

## Открытые задачи (по убыванию приоритета)

### 🔴 P0 — критично

1. **Архитектурный фикс `WatchRecordsTask`** — в процессе в отдельной сессии. Без него следующий transient PG/IO error снова уронит writer молча. Промпт: [docs/tasks/watch-records-task-supervisor-prompt.md](../tasks/watch-records-task-supervisor-prompt.md).

### 🟡 P1 — превентивные меры (на этой неделе)

2. **`pg_checksums --enable`** на frigate_analyzer cluster. Без этого PG не ловит мусорные страницы от FS — следующий аппаратный сбой может опять привести к молчаливому corruption. Требует останова PG:
   ```bash
   sudo systemctl stop postgresql@17-main
   sudo -u postgres /usr/lib/postgresql/17/bin/pg_checksums --enable \
     -D /var/lib/postgresql/17/main
   # ~15-20 min на 350 GB
   sudo systemctl start postgresql@17-main
   ```
   На время остановки лягут также filer-БД (~15-20 минут downtime для seaweedfs filer-ов).

3. **Cron еженедельный `btrfs scrub start /`** — чтобы латентная corruption не накапливалась годами как было до 23 мая. Пример:
   ```
   # /etc/cron.d/btrfs-scrub
   0 3 * * 0 root /usr/bin/btrfs scrub start -B /     >> /var/log/btrfs-scrub.log 2>&1
   0 4 * * 0 root /usr/bin/btrfs scrub start -B /boot >> /var/log/btrfs-scrub.log 2>&1
   ```

4. **Cron еженедельный `pg_amcheck`** — чтобы corruption ловить рано, до того как INSERT упадёт:
   ```
   0 5 * * 0 postgres /usr/lib/postgresql/17/bin/pg_amcheck -d frigate_analyzer --heapallindexed -j 2 \
                                                            >> /var/log/pg-amcheck.log 2>&1
   ```

5. **Алёрт на ошибки PG в логе.** journald → промтейл/loki → grafana alert на:
   - `XX000`, `XX002`
   - `could not access status of transaction`
   - `uncommitted xmin needs to be frozen`
   - `Segmentation fault`
   - `terminating any other active server processes`

6. **Алёрт на `ata1: SError` в kernel-логе.** Аналогично — `journalctl -u kernel | grep ata.*SError` → alert. Даст ранее предупреждение, если кабель снова заиграет.

### 🟢 P2 — мелкие улучшения

7. **Owner auto-bootstrap.** Открытый bug, выявлен сегодня: после drop database owner не может пользоваться `/timezone`/`/notifications` пока не отправит `/start`. Это плохой UX (silent fail с WARN в логе). Варианты фикса:
   - Liquibase changeset, который читает `TELEGRAM_OWNER` env var на старте (через app, не SQL) и делает upsert
   - Lazy auto-create в `TelegramUserServiceImpl.getUserZone()` / `.updateTimezone()` если username совпадает с `properties.owner`
   - `ApplicationListener<ApplicationReadyEvent>` который при старте делает `ensureOwnerExists()`

8. **Detect server `mypc` в конфиге.** Указан в `application-docker.yaml`, но хост выключен — постоянно DEAD. Убрать из конфига или закомментировать, чтобы не пугать статистику.

9. **`vps` detect server flapping.** За 23 мая было 93×DEAD/93×ALIVE — сетевые таймауты к удалённому хосту. Не критично, но шумит в логе. Возможно подкрутить таймаут health-check или сменить транспорт.
