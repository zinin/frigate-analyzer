# Merged Design Review — Iteration 1

## codex-executor (gpt-5.3-codex)

### Critical Issues
1. Несогласованная семантика даты в `/export`. В design заявлено, что ввод интерпретируется в TZ пользователя, но в плане не зафиксировано, как считать `Сегодня/Вчера`; в реализации это серверный `clock`. Это даёт неправильный день для пользователей в другом часовом поясе.
2. Неполная модель ошибок для ручного ввода TZ. Документы фиксируют только `ZoneRulesException`, но `ZoneId.of(...)` может кидать и другие `DateTimeException`. Это риск падения сценария `/timezone`.
3. В плане нет тестовой стратегии для ключевой логики времени.
4. Нет защиты от невалидного `olson_code` в БД. В сервисе это прямой `ZoneId.of(...)`, что может сорвать рассылку всем пользователям.

### Concerns
1. Для `/timezone` не описан timeout/cleanup waiter'ов.
2. `INTERVAL '10 seconds'` в SQL не объяснён как бизнес-правило в design.
3. Хардкод 8 зон СНГ ограничивает UX и масштабирование на международных пользователей.
4. `updateTimezone` в плане не проверяет affected rows.

### Suggestions
1. Явно зафиксировать правило: `today/yesterday` вычислять в зоне пользователя.
2. Обновить error-handling: ловить `DateTimeException`, давать retry, и иметь safe-fallback при чтении TZ из БД.
3. Добавить обязательные тесты в план.
4. Рассмотреть архитектурно отдельный `TimezoneService`.
5. Как альтернатива текущему nullable-подходу: `olson_code NOT NULL DEFAULT 'UTC'` + backfill.

### Questions
1. `Сегодня/Вчера` должны считаться по TZ пользователя или по TZ сервера?
2. Как обрабатывать DST-случаи (несуществующее/двусмысленное локальное время) в `/export`?
3. Что делать при невалидном `olson_code` в БД: fallback на UTC, авто-сброс, или блокировка пользователя с уведомлением?
4. Нужен ли timeout для `/timezone` waiter (аналогично `/export`)?
5. Нужно ли формально задокументировать смысл `-10 seconds` на старте диапазона и вынести его в конфиг?

## gemini-executor

### Critical Issues
Отсутствуют — план выглядит целостным и не содержит блокирующих архитектурных ошибок.

### Concerns
1. Валидация ручного ввода и DST: `ZoneId.of()` принимает фиксированные смещения (UTC+03:00, GMT+2). Пользователь потеряет автоматический переход на летнее/зимнее время.
2. Масштабируемость уведомлений: отправка происходит последовательно в цикле `usersWithZones.forEach`.
3. Возврат `Pair<Long, ZoneId>` — "code smell", невыразительная сигнатура.

### Suggestions
1. Улучшение валидации: проверка наличия `/` в введенном ID.
2. DTO для сервиса: создать `UserZoneInfo(val chatId: Long, val zone: ZoneId)`.
3. Тесты: добавить unit-тесты для конвертации времени и форматирования.
4. Индексация: убедиться, что `record_timestamp` имеет индекс (подтверждено — есть).

### Questions
1. Буфер 10 секунд: новая логика или перенос существующей?
2. Формат даты в уведомлениях: планируется ли изменение паттерна?

## ccs-executor (glmt)

### Critical Issues
Отсутствуют.

### Concerns
1. Отсутствие валидации на уровне сервиса: `TelegramUserServiceImpl.updateTimezone` не валидирует Olson ID.
2. Имя файла экспорта зависит от часового пояса пользователя — затрудняет отладку.
3. Отсутствие логирования предыдущего значения часового пояса.
4. Жёстко заданный список часовых поясов.

### Suggestions
1. Добавить команду `/timezone` для просмотра без изменения.
2. Локаль всегда русская, независимо от timezone — документировать.
3. Потенциальная N+1 проблема при уведомлениях.
4. Отсутствие обработки специальных Olson ID (GMT+3 и подобные).

### Questions
1. Почему default UTC вместо Moscow?
2. Планируется ли поддержка нескольких часовых поясов для одного пользователя?
3. Почему `VARCHAR(50)` когда максимальный Olson ID ~30 символов?
4. Учитывается ли timezone при отладке?
