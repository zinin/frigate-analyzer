# Telegram `/export` — режим `original|annotated` и прогресс

## Контекст

Текущая команда `/export` выгружает только оригинальное видео (склейка сегментов, при необходимости компрессия), затем отправляет файл в Telegram.

Ключевые точки текущей реализации:
- Диалог и отправка видео: `modules/telegram/src/main/kotlin/ru/zinin/frigate/analyzer/telegram/bot/FrigateAnalyzerBot.kt`
- Экспорт оригинального видео: `modules/core/src/main/kotlin/ru/zinin/frigate/analyzer/core/service/VideoExportServiceImpl.kt`
- Аннотация видео (async submit/poll/download): `modules/core/src/main/kotlin/ru/zinin/frigate/analyzer/core/service/VideoVisualizationService.kt`

## Цель

Расширить `/export` так, чтобы пользователь мог выбрать:
- `Оригинал`
- `С объектами`

И показывать прогресс в виде **одного обновляемого сообщения** в чате:
- для `Оригинал`: этапы обработки;
- для `С объектами`: этапы + проценты аннотации.

## Подтверждённые решения

1. Выбор режима — **кнопками в диалоге** (не аргументом команды).
2. Прогресс — **для обоих режимов**.
3. UX прогресса — **одно редактируемое сообщение**.
4. В `С объектами` пока без дополнительных пользовательских параметров.
5. Если аннотация упала — **ошибка и завершение** (без авто-fallback на оригинал).
6. Для API аннотации:
   - `classes` берём из `application.detection-filter.allowed-classes`;
   - `model` берём из `application.detect.good-model`.

## Область изменений

### 1) Telegram dialog (`/export`)

В `handleExport(...)` после выбора камеры добавить шаг выбора режима:
- `Оригинал` → `ExportMode.ORIGINAL`
- `С объектами` → `ExportMode.ANNOTATED`

Текущий порядок станет:
1. Дата
2. Диапазон времени
3. Камера
4. **Режим экспорта**
5. Выполнение и прогресс

### 2) Контракт сервиса экспорта

Расширить `VideoExportService`:
- добавить `mode: ExportMode`;
- добавить callback прогресса `onProgress`.

Сервис возвращает итоговый `Path` готового файла (оригинал или аннотированный) и сохраняет ответственность за очистку временных артефактов, кроме финального файла, который удаляется в `cleanupExportFile(...)` после отправки.

### 3) Реализация экспорта в core

`VideoExportServiceImpl`:
- `ORIGINAL`: как сейчас (поиск файлов → merge → optional compress → return path).
- `ANNOTATED`:
  1. выполнить `ORIGINAL`-часть и получить промежуточный mp4;
  2. вызвать `VideoVisualizationService.annotateVideo(...)`;
  3. передать:
     - `classes = allowedClassesCsv` из `DetectionFilterProperties.allowedClasses`;
     - `model = DetectProperties.goodModel`;
  4. удалить промежуточный файл, вернуть путь аннотированного файла.

## Модель прогресса

Вводится событийная модель прогресса с этапами.

### Этапы

- `PREPARING` — поиск записей и проверка наличия файлов
- `MERGING` — склейка видео
- `COMPRESSING` — компрессия (если нужна)
- `ANNOTATING` — удалённая аннотация (только `ANNOTATED`)
- `SENDING` — отправка файла в Telegram
- `DONE` — завершено

Для `ANNOTATING` дополнительно передаётся процент (`0..100`) из `JobStatusResponse.progress`.

### Рендер в Telegram

Одно сообщение, редактируемое по мере прогресса:
- этапы без реального `%` — текстовый статус + условная шкала;
- этап `ANNOTATING` — фактический процент из polling.

Ограничение частоты обновлений:
- обновлять при смене этапа;
- и/или при изменении процента аннотации на заметный шаг (например, 5%).

Если редактирование статуса не удалось, это логируется как warning и не прерывает экспорт.

## Поток выполнения

### ORIGINAL

1. Бот отправляет стартовое статус-сообщение.
2. `PREPARING` → `MERGING` → (`COMPRESSING` при необходимости).
3. Возвращается итоговый файл.
4. Бот переключает статус на `SENDING` и отправляет видео.
5. Очистка временного файла.
6. `DONE`.

### ANNOTATED

1. Шаги `PREPARING`/`MERGING`/`COMPRESSING` как в `ORIGINAL`.
2. `ANNOTATING`:
   - submit job;
   - polling статуса;
   - download результата;
   - прогресс в чат из `JobStatusResponse.progress`.
3. Бот: `SENDING` и отправка аннотированного видео.
4. Очистка временных файлов (включая промежуточный merged/compressed файл).
5. `DONE`.

## Ошибки и таймауты

### Ошибки original-части

- Нет записей / отсутствуют файлы / итог всё ещё слишком большой — сообщаем ошибку экспорта.

### Ошибки annotated-части

- `DetectTimeoutException`, `VideoAnnotationFailedException`, HTTP-ошибки и другие сбои аннотации:
  - сообщаем, что экспорт с объектами не выполнен;
  - **не** отправляем оригинал автоматически;
  - выполняем cleanup.

### Таймауты

- Сохраняется общий timeout обработки команды в боте.
- Для аннотации учитывается timeout `application.detect.video-visualize.timeout`.
- Настройка timeout-ов должна исключать ситуацию, когда бот обрывает задачу раньше ожидаемого таймаута аннотации.

## Тестирование

### Unit: `VideoExportServiceImpl`

Обязательные сценарии:
1. `ORIGINAL` happy path
2. `ORIGINAL` + compress
3. нет записей
4. все файлы отсутствуют
5. `ANNOTATED` happy path
6. `ANNOTATED` error path + cleanup промежуточных файлов
7. порядок и содержание progress-событий

### Unit/flow: Telegram `/export`

Сценарии:
1. после выбора камеры появляется выбор режима;
2. выбран `ORIGINAL` → вызывается экспорт в ORIGINAL;
3. выбран `ANNOTATED` → вызывается экспорт в ANNOTATED;
4. прогресс отображается через одно редактируемое сообщение;
5. ошибка аннотации завершает сценарий без fallback.

## Критерии готовности

1. В `/export` добавлен шаг выбора `Оригинал` / `С объектами`.
2. Оба режима корректно отправляют соответствующий файл.
3. Прогресс для обоих режимов отображается одним обновляемым сообщением.
4. Для аннотации используются:
   - `detection-filter.allowed-classes`;
   - `detect.good-model`.
5. На ошибке аннотации нет авто-перехода на оригинал.
6. Добавленные тесты проходят.

## Out of scope

- Добавление пользовательских параметров аннотации (`conf`, `classes`, `model`) в UI.
- Рефакторинг бота в отдельный оркестратор (можно рассмотреть позже при росте сложности).
