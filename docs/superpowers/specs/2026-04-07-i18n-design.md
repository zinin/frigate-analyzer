# I18n Support for Frigate Analyzer Telegram Bot

## Overview

Add multi-language support (Russian + English) to the Telegram bot with automatic language detection and manual override. Migrate all code documentation to English.

## Goals

- Extract ~120 hardcoded Russian user-facing strings into Spring MessageSource `.properties` files
- Support `ru` and `en` with infrastructure ready for more languages
- Auto-detect language from Telegram `language_code` at registration
- Allow manual language switch via `/language` command
- Personalize notifications per user's language preference
- Migrate all KDoc, inline comments, and OpenAPI annotations to English

## Non-Goals

- Localization of application logs
- Localization of Swagger UI itself (only OpenAPI annotation descriptions)
- Support for languages beyond `ru`/`en` (infrastructure supports it, but only two language files)
- Pluralization support (no complex plural forms in current messages)

---

## 1. User Language Storage

### Database

New column in `telegram_users`:

```sql
ALTER TABLE telegram_users ADD COLUMN language_code VARCHAR(64) NOT NULL DEFAULT 'ru';
```

Liquibase migration in `docker/liquibase/migration/`.

### Language Detection at `/start`

When a user sends `/start`:
1. Read `message.from.language_code` from Telegram (IETF tag, e.g. `en`, `ru`, `en-US`)
2. If it starts with `en` вЖТ save `en`, otherwise вЖТ save `ru` (fallback to Russian)
3. Store in `language_code` column on user creation/activation

### `/language` Command

- Displays inline keyboard with buttons: `рЯЗЈрЯЗЇ –†—Г—Б—Б–Ї–Є–є`, `рЯЗђрЯЗІ English`
- Updates `language_code` in the database
- Confirms in the newly selected language (e.g. "Language set to English")
- Owner-only: no вАФ available to all authorized users

### DTO Changes

Extend `UserZoneInfo` with a `language` field:

```kotlin
data class UserZoneInfo(
    val chatId: Long,
    val zone: ZoneId,
    val language: String,  // "ru" or "en"
)
```

This DTO is used in both notification sending and command handlers.

### MessageResolver Service

```kotlin
@Component
class MessageResolver(private val messageSource: MessageSource) {

    fun get(key: String, language: String, vararg args: Any): String =
        get(key, Locale.forLanguageTag(language), *args)

    fun get(key: String, locale: Locale, vararg args: Any): String =
        try {
            messageSource.getMessage(key, args, locale)
        } catch (e: NoSuchMessageException) {
            logger.warn { "Missing translation key='$key' for locale='$locale'" }
            try {
                messageSource.getMessage(key, args, Locale.forLanguageTag("ru"))
            } catch (e: NoSuchMessageException) {
                key
            }
        }
}
```

Fallback chain: requested locale вЖТ Russian вЖТ raw key. Injected into all command handlers and notification services.

### Spring Configuration

Standard `MessageSource` bean configured via `ReloadableResourceBundleMessageSource` with `UTF-8` encoding and basename `classpath:messages`.

## 3. Integration Points

### Command Handlers

Each handler receives `MessageResolver` via constructor injection.

`CommandHandler.handle()` interface receives `TelegramUserDto?` instead of `UserRole?`. The DTO is resolved once during authorization in `FrigateAnalyzerBot.registerRoutes()` вАФ no duplicate DB queries.

Flow:
1. `FrigateAnalyzerBot` resolves user DTO during authorization (already fetching user for role check)
2. Passes `TelegramUserDto?` to `handler.handle(message, user)`
3. Handler reads `user?.languageCode ?: detectLanguage(telegramLangCode)` вАФ zero extra DB queries
4. For unauthorized users (before `/start`): `user` is `null`, fallback from Telegram's `message.from.language_code`

### Notification Service (`TelegramNotificationServiceImpl`)

Currently `getAuthorizedUsersWithZones()` returns `List<UserZoneInfo(chatId, zone)>`. Extended to include `language`.

Per-user notification loop:
- Each notification formatted individually with the recipient's `Locale`
- `DateTimeFormatter` receives the correct `Locale` (currently hardcoded to `ru`)

### Quick Export Buttons

Button labels and progress states are formatted in the language of the user who initiated the export.

### Bot Command Descriptions

Use Telegram Bot API `setMyCommands` with `language_code` parameter to register command descriptions for both `ru` and `en`. Telegram displays the description matching the user's interface language.
